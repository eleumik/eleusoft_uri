116     * <p><strong>Percent Encoding - "URI Escaping"</strong>
117     * <p>An URI is <em>always in an percent encoded form</em>.
118     * URI escaping is the old term for <em>percent encoding</em>.
119     * <p>
120     * <blockquote>
121     *  <p>From RFC3986 Section 2.1 - Percent Encoding
122     * <p><cite>
123       A percent-encoding mechanism is used to represent a data octet in a
124       component when that octet's corresponding character is outside the
125       allowed set or is being used as a delimiter of, or within, the
126       component.  A percent-encoded octet is encoded as a character
127       triplet, consisting of the percent character "%" followed by the two
128       hexadecimal digits representing that octet's numeric value.  For
129       example, "%20" is the percent-encoding for the binary octet
130       "00100000" (ABNF: %x20), which in US-ASCII corresponds to the space
131       character (SP).</cite>
132     * </blockquote>
133     * Being alwyays in escaped form means that
134     * <em>decoding the percent escaped sequences</em>
135     * of an URI
136     * <em>may not preserve the information present in the URI</em>.
137    
138     * <p>RFC 3986 allows percent-encoded octets to appear in
139     * the user-info, path, query, and fragment components.
140     * Escaping serves two purposes in URIs: to insert
141     * non-ASCII characters in the URI and to insert
142     * URI-reserved characters inside the URI avoiding that
143     * their interpretation as delimiters
144     * for URI components and sub components. Not all the
145     * URI-reserved characters are actually used as delimiters,
146     * a basic set is always reserved for each component,
147     * while some are free for use by URI scheme implementations
148     * (for designers of an URI scheme).
149     * <p>
150     * <blockquote>
151     *  <p>From RFC3986 2.2 -  Reserved Characters
152     * <p><cite>
153       URIs include components and subcomponents that are delimited by
154       characters in the "reserved" set.  These characters are called
155       "reserved" because they may (or may not) be defined as delimiters by
156       the generic syntax, by each scheme-specific syntax, or by the
157       implementation-specific syntax of a URI's dereferencing algorithm.</cite>
158     * </blockquote>
159     * <p>Reserved characters are divided in general delimiters and
160     * sub delimiters.
161     *
162     * <pre class=code>
163     *
164     * reserved    = gen-delims / sub-delims
165     *
166     * gen-delims  = ":" / "/" / "?" / "#" / "[" / "]" / "@"
167     *
168     * sub-delims  = "!" / "$" / "&amp;" / "'" / "(" / ")"
169     *                  / "*" / "+" / "," / ";" / "="
170     *
171     * unreserved  = ALPHA / DIGIT / "-" / "." / "_" / "~"
172     * pct-encoded = "%" HEXDIG HEXDIG
173     * </pre>
174     *
175     * <p>General delimiters in the first component that
176     * support percent escaped octets (user-info) must always
177     * be escaped. In the following percent-escaping-supporting
178     * component, path, all general delimiters
179     * but ':' and '@' must be escaped (':' has an exception [**] when
180     * is in the first segment of a relative reference),
181     * in the other components also '?' and '/' can be present.

328     * be escaped to represent its value in each component decrements when you move
329     * from the first (user-info) to the last (fragment).
330     * <blockquote>
331     *  <p>From RFC3986 Section 2.4 - When to Encode or Decode
332     *  <p><cite>
333     *   Under normal circumstances, the only time when octets within a URI
334     *  are percent-encoded is during the process of producing the URI from
335     * its component parts.  This is when an implementation determines which
336     * of the reserved characters are to be used as subcomponent delimiters
337     * and which can be safely used as data.  Once produced, a URI is always
338     * in its percent-encoded form</cite>
339     * </blockquote>
340     * <p><b>URI percent encoding example</b>
341     *  <p>This is example may help in understanding
342     * why is necessary to escape an URI.
343     * Suppose we receive two URIs like these:
344     * <ol>
345     * <li> &lt;&lt;<code>http://www.example.org/Number#A1.xml</code>&gt;&gt;
346     * <li> &lt;&lt;<code>http://www.example.org/Number<em>%23</em>A1.xml</code>&gt;&gt;
347     * </ol>
348     * <p>While both have the same scheme <code>http</code>,
349     * the same authority <code>www.example.org</code>,
350     * and absent query, the first URI has path
351     * <code>&lt;&lt;/Number>></code>
352     * and fragment <code>&lt;&lt;A1.xml>></code> while
353     * the second one has path
354     * <code>&lt;&lt;/Number<em>%23</em>A1.xml>></code>
355     * and <em>absent</em> fragment.
356     * In the first case an URI parser will interpret
357     * the hash '#' symbol as the URI-reserved-control-char
358     * that marks the begin of an URI fragment component,
359     * In the latter case the '#' is encoded as '%23' so
360     * that will be interpreted as part of the path;
361     * in this second case it might be possible for a web server
362     * to map the URI to a file with <em>file</em>-name
363     * <code>Number#A1.xml</code> on the web server machine,
364     * also if whether this actually happens will
365     * depend on how the web server is configured.
366     * In the first case instead the
367     * file-name would have been <code>Number</code>.
368     *
369     * <p><strong>URI Encoding character set (browsers)</strong>
370     * <p>Some browsers URLs to be written inside HTML documents
371     * using characters belonging to the HTML document charset
372     * without requiring percent encoding.
373     * The result is a more readable page and the automatic correction
374     * of wrongly typed URIs. It is instead necessary to
375     * always encode the url using <emASCII</em>,
376     * percent encoding the non-ASCII characters.
377     * In this way  we can insert in an HTML page
378     * encoded with standard 8 bit western european encoding (iso-8859-1)
379     * an HTML link containing and URL that contains japanese characters.
380     * Cfr. <a href='http://www.w3.org/TR/html401/appendix/notes.html#non-ascii-chars'>
381     * http://www.w3.org/TR/html401/appendix/notes.html#non-ascii-chars</a>.
382     * <p>Browsers in general are very forgiving about URI syntax.

399     *      <p><strong>URI and PATH</strong>
400     *      <p>A path is always defined for a URI, though the
401     *      defined path may be empty (zero length?).
402     *      <p>Path is a common concept for file systems.
403     *      File system paths are not normally percent-encoded and they simply
404     *      do not allow some characters inside path expressions.
405     *      <p class='note'>Windows/DOS file/folder names do not allow the following
406     *      9 characters:<code>\ / ? : * &quot; &lt; | &gt; </code>
407     *      in windows explorer names that start or end with the char <code>.</code> are not allowed,
408     *              [msg:You must type a file name] in command line they are allowed.
409     *      For LINUX all characters are allowed apart [.] [..] [ ] and [/]
410     *      <p>This is almost the same of what happens for the URI path
411     *      component also if the URI path theoretically allows more characters
412     *      (that are anyway better not to use since they cannot find place on file systems).
413     *              This interface exposes the path also in unescaped form, this might be
414     *              wrong but is what most of the API do.
415     *      <p><strong>query component</strong>
416     *
417     *      <p>Some implementations of URI make available the query in unescaped form.
418     *      This for me is wrong since the query might loose of meaning in the process of
419     *      unescaping. See an example in the {@link #getQuery()} method.
420     *      <p>See also discussion <a href='http://lists.w3.org/Archives/Public/uri/2006Jan/0025.html'>
421     *      http://lists.w3.org/Archives/Public/uri/2006Jan/0025.html</a>.
422     *      <p>Query must be opaque since its meaning depends on the scheme.<ul>
423     *              <li>There is no standard about query internal structure,
424     *              <li>separator for parameters might be [&amp;] or [;].
425     *      <li>Parameters could be specified also using application/x-www-form-urlencoded
426     *      so [+] in place of [%20] etc.
427     *              </ul>
428     *              <p>Note on + and %20 : When sent in an HTTP GET request,
429     *              application/x-www-form-urlencoded data is included
430     *              in the query component of the request URI
431     *              (When sent in an HTTP POST request or via email,
432     *              the data is placed in the body of the message,
433     *              and the name of the media type is included in the message's Content-Type header),
434     *              since the x-www-form-urlencoded  format allows + as placeholder
435     *              (this should be syntactically ok since + is reserved in the query),
436     *              for spaces inside the query it is only a x-www-form-urlencoded format parser that
437     *              provided of the query component (percent encoded of course),
438     *              will be able to decode the + as a space.

441     *      <p><strong>URI path resolving / normalization - meaningless path</strong>
442     *      <p>Some implementations of URI (java.net.URI) resolve and normalize paths that attempt
443     *      to reach files before the root making available in the resolved or normalized
444     *      URI a path that does not   mean anything like "/../../".
445     *      <p>In this cases the beahvior of implementations of this interface must be to return
446     *      empty string as path after normalization.
447     *      <p>In general, after that resolution brings an empty path,
448     *      checking that the path component of the relative URI passed
449     *      to {@link #resolve(URI)} method is not empty is enough to determine that
450     *      resolution brought an invalid result.
451     *      <p><strong>URI null and empty string path.</strong>
452     *      <p>An empty string path can be present because an  uri has been created with no path
453     *       <ul><p><em>Example</em>:
454     *      <li><code>http://www.example.org</code>
455     *      <li><code>http://www.example.org#fds</code>
456     *      <li><code>#fds</code>
457     *      <li><code>?param=value</code></ul>
458     *      <p>But also because the URI after normalization contains an meaningless path:
459     *       <ul><p><em>Example</em>:
460     *      <li><code>http://www.example.org/../../file.xml</code>
461     *      <li><code>http://www.example.org/s/../../../p.xml#fds</code></ul>
462     *      <p>In my opinion after normalization a meaningless path might
463     *      in some cases become the root path (if the URI is not opaque and is absolute),
464     *      since <code>http://www.example.org/</code>
465     *      and <code>http://www.example.org</code> are commonly considered the same,
466     *              but again this depends on the uri schema.

502     * <p>One of the scopes of this package is to
503     * provide an URI implementation of not
504     * based on the JDK(1.4+) <a href='http://java.sun.com/j2se/1.4.2/docs/api/java/net/URI.html'>

506     * reasons and because that class has been created before RFC3986
507     * and has some inconsistent behavior respect to that specification.
508     * At the moment exist two implementations, one based on JDK1.4+
509     * <code>java.net.URI</code> and an other one based on
510     * the <code>org.apache.util.URI</code> class, both wrapped by
511     * a 'correction' adaptor.
512     * <p>Test cases have been produced using some uri tests and examples available
513     * on the web. They are available here [link to be added TODO].
514     *
515     * <p>References
516     * <ol>
517    * <li><a href='http://www.ietf.org/rfc/rfc3986.txt'>
518     * http://www.ietf.org/rfc/rfc3986.txt</a> (text)
519     * <a target='_blank' href='http://www.ietf.org/rfc/rfc3986.txt'>
520     * [new window]</a>
521     * <li><a href='http://www.apps.ietf.org/rfc/rfc3986.html'>

524     * [new window]</a>

529     * <li><a href='http://www.w3.org/Addressing/URL/4_URI_Recommentations.html'>

537        /**
538         * <em>Convenience method</em> that
539         * returns whether the URI is
540         * absolute (has a scheme component).
541         * <p>Since an implementation of this
542         * interface cannot return an empty
543         * string from {@link #getScheme()),
544         * this method is equivalent to:
545         * <pre class=code>
546         *     return scheme!=null;
547         * </pre>
548         * <p>Note that in RFC Section 4.3
549         * <a href='http://www.apps.ietf.org/rfc/rfc3986.html#sec-4.3'>
550         * http://www.apps.ietf.org/rfc/rfc3986.html#sec-4.3</a>
551         * an absolute uri is specified as also
552         * having no fragment.
553         * <blockquote>
554         * Some protocol elements allow only the absolute form of a URI without
555         * a fragment identifier. For example, defining a base URI for later
556         * use by relative references calls for an absolute-URI syntax rule
557         * that does not allow a fragment.
558         * </blockquote>
559         * <p>Here instead only the scheme presence is tested.
560         */
561        boolean isAbsolute();
562    
563        /**
564         * <em>Convenience method</em> that
565         * returns whether the URI is a relative URI
566         * and has as path component a relative
567         * path, that is a path that is empty
568         * or does not start with the char <code>/</code>.
569         * <p>This method is equivalent to:
570         * <pre class=code>
571         * !uri.isAbsolute() &&
572         *      (uri.isPathEmpty() || uri.getPEPath().charAt(0)!='/')
573         * </pre>
574         */
575        boolean isRelativePath();
576    
577    
578        /**
579         * Returns the <code>scheme</code>
580         * component of the URI.
581         * <p>
582         * <pre class=code>
583         * scheme      = ALPHA *( ALPHA / DIGIT / "+" / "-" / "." )
584         * </pre>
585         * <p>The URI <code>scheme</code>
586         * cannot be empty, it is either absent
587         * or with a non-empty value
588         * so this method must never return an empty
589         * string.
590         * <p>The URI <code>scheme</code> cannot
591         * contain percent encoded characters
592         * so there is no difference between the
593         * two forms.
594         * <p>The URI <code>scheme</code> is case - unsensitive,
595         * upper case scheme are accepted
596         * but this method returns always a lowercase
597         * string.
598         * <blockquote>
599         * FROM RFC3986 - Section 3.1 - Scheme
600         * <p><cite>Although schemes are case-
601         *  insensitive, the canonical form is lowercase
602         * and documents that  specify schemes must do so
603         * with lowercase letters.  An implementation
604         * should accept uppercase letters as equivalent
605         * to lowercase in scheme names
606         * (e.g., allow "HTTP" as well as "http")
607         * for the sake of  robustness but should
608         * only produce lowercase scheme names for
609         * consistency.</cite>
610         * </blockquote>
611         * @return the lower case value of the scheme
612         * component or null when the scheme is absent,
613         * never empty string.
614         */
615        String getScheme();
616    
617        /**
618         * Tells whether or not this URI is opaque.
619         * An opaque URI is not parsed beyond the scheme component.
620         * When the uri is opaque its scheme specific part
621         * can be retrieved using {@link #getSchemeSpecificPart()}
622         */
623    
624        public boolean isOpaque();
625    
626        /**
627         * Returns the decoded scheme-specific
628         * part of this URI when opaque.
629         */
630        public String getSchemeSpecificPart();

632    
633

635         * Returns the <code>authority</code>
636         * component of the URI.
637         * The authority value returned by
638         * this method is unescaped.
639         */
640        String getAuthority();

645             * path component of the uri.
646             */
647            String getPDPath();
648            /**
649             * Retrieves the <b>p</b>ercent <b>e</b>ncoded
650             * path component of the uri.
651             */
652            String getPEPath();
653            /**
654             * Signals whether the path component
655             * is empty.
656             * <p>
657             * <blockquote class=code>
658             * <p>RFC3986 Section 3.3 - Path
659             * <p><cite>
660             * <pre>A path is always defined for a URI, though the
661         * defined path may be empty (zero length).
662         * Use of the slash character
663         * to indicate hierarchy is only required when a URI will be used as the
664         * context for relative references.  For example, the URI
665         * &lt;mailto:fred@example.com> has a path of "fred@example.com", whereas
666         * the URI &lt;foo://info.example.com?fred> has an empty path.</cite>
667         * </blockquote>
668         */
669        boolean isPathEmpty();
670    
671            /**
672             * Signals whether the path component
673             * does not start with slash '/' character.
674             * <p>
675             * <blockquote class=code>
676             * <p>RFC3986 Section 3.3 - Path
677             * <p><cite>
678             * <pre>path-rootless = segment-nz *( "/" segment )</pre></cite>
679         * </blockquote>
680         * <p>For example the URI
681         * &lt;mailto:fred@example.com> has a path of "fred@example.com"
682         * and so is root less.
683         * <p>When the result of this method is anded together
684         * with the result of {@link #isAbsolute()},
685         * it generate the result of {@link #isOpaque()}
686         */
687        boolean isPathRootless();
688    
689        /**
690         * Returns the percent encoded <code>query</code>
691         * component of the URI.
692         * <pre class=code>
693             * query = *( pchar / "/" / "?" )
694         * </pre>
695         * <p>The query is always percent encoded,
696         * since subsequent decoding and encoding
697         * might cause data lost.
698         * <p>Example of data loss (param value containing & symbol):
699         * <p><pre class=code>
700         * URI: http://127.0.0.1:8080/Test.jsp?current=R%26D
701         * QUERY param name: current
702         * QUERY param value: R&D
703         * UNESCAPED QUERY: current=R&D
704         * ESCAPED QUERY: current=R&D
705         *          !!! error:
706         *          [char '&' is interpreted as a query param delimiter]
707         * </pre>
708         * @return the percent encoded query string value
709         * or null when no query component.
710         * Might return the empty string.
711         */
712        String getQuery();
713    
714        /**
715         * <em>Convenience method</em> that
716         * returns whether the
717         * <code>query</code> is <em>absent</em>.
718         * <p>Equivalent to:
719         * <pre class=code>
720         * uri.getQuery()!=null
721         * </pre>
722         * <p>When the query is empty
723         * (a zero length string)
724         * this method returns true.
725         * <p>An empty query is built for
726         * example from an uri like the following:
727         * &lt;&lt;<code>http://www.example.com/index.jsp?</code>&gt;&gt;
728         *
729         */
730        public boolean hasQuery();
731    
732        /**
733         * Returns the <code>fragment</code>
734         * component of the URI.
735         * The path value returned by
736         * this method is unescaped.
737         */
738        String getFragment();
739    
740            /**
741             * Returns the percent encoded fragment.
742             * <pre class=code>
743             * fragment    = *( pchar / "/" / "?" )
744         * </pre>
745             * <blockquote class=code>
746             * <p>RFC3986 Section 3.5 - Fragment
747             * <p><cite>
748         * The fragment identifier component of a URI allows indirect
749         * identification of a secondary resource by reference to a
750         * primary resource and additional identifying information.
751         * The identified secondary resource may be some portion or subset
752         * of the primary resource, some view on representations of
753         * the primary resource, or some other resource defined or
754         * described by those representations. A fragment identifier
755         * component is indicated by the presence of a number sign
756         * ("#") character and terminated by the end of the URI.
757         * @return the percent encoded fragment or null when no fragment.
758         * Might return the empty string.
759             */
760        String getPEFragment();
761        /**
762         * <em>Convenience method</em> that
763         * returns whether the
764         * fragment is <em>absent</em>.
765         * <p>Equivalent to:
766         * <pre class=code>
767         * uri.getFragment()!=null
768         * </pre>
769         * <p>When the fragment is empty
770         * (a zero length string)
771         * this method returns true.
772         * <p>An empty fragment is built for
773         * example from an uri like the following:
774         * &lt;&lt;<code>http://www.example.com/index.jsp#</code>&gt;&gt;
775         */
776        boolean hasFragment();
777        /**
778         * Returns a readable URI.
779         * The returned string
780         * can be only used
781         * for presentation scopes
782         * since it is possible that
783         * is not anymore parsable
784         * by a {@link URIFactory}.
785         */
786        String toReadableURI();

790         * of an URI in percent encoded form.

825        /**
826         * Normalizes this URI's path.
827         * <p>When in the relative path there are more
828         *  relative path ".." segments than there are hierarchical levels in the
829         *  base URI's path this method returns an URI with <em>empty path</em>.
830         * <p>See  RFC 2396, section 5.2, step 6, sub-steps c through f;
831         * <blockquote><code>from Appendix C.2. "Abnormal Examples"</code>
832         *  <p>Parsers must be careful in handling the case where there are more
833         *  relative path ".." segments than there are hierarchical levels in the
834         *  base URI's path.  Note that the ".." syntax cannot be used to change
835         *  the authority component of a URI.
836         *  <pre class=code>
837         *  BASE URI: http://a/b/c/d;p?q
838         *
839         *  ../../../g    =  http://a/../g
840         *  ../../../../g =  http://a/../../g
841         *  </pre>
842         *  <p>In practice, some implementations strip leading relative symbolic
843         * elements (".", "..") after applying a relative URI calculation, based
844         * on the theory that compensating for obvious author errors is better
845         * than allowing the request to fail.  Thus, the above two references
846         * will be interpreted as "http://a/g" by some implementations.
847         * </blockquote>
848         * <p>Implementations of this interface won't return <code>http://a/g</code>
849         * in this case but <code>http://a</code>
850         * <blockquote>
851         * <p>Continuing from  RFC 2396
852         * <p>Similarly, parsers must avoid treating "." and ".." as special when
853         *  they are not complete components of a relative path.
854         * <pre class=code>
855         *  /./g          =  http://a/./g
856         *  /../g         =  http://a/../g
857         *  g.            =  http://a/b/c/g.
858         *  .g            =  http://a/b/c/.g
859         *  g..           =  http://a/b/c/g..
860         *  ..g           =  http://a/b/c/..g
861         * </pre>
862         * </blockquote>
863         * <p>Implementations of this interface must give results that conform
864         * to this last list of examples.
865         * <p>See also class description.
866         */
867        URI normalize() throws URIException;

871         * @throws IllegalArgumentException - If this URL is not absolute
872         * @throws MalformedURLException - If a protocol handler for the URL
873         *      could not be found, or if some other error occurred
874         *      while constructing the URL
875         * @return the URL instance, never null.
876         */
877        java.net.URL toURL() throws java.net.MalformedURLException;
